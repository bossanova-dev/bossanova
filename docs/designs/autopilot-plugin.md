## Vision

### 10x Check

The 10x version of autopilot isn't just "run skills in sequence" — it's an AI project manager that understands the development lifecycle end-to-end. It would:

1. **Adaptive flight planning** — instead of a fixed skill sequence, the plugin observes each flight leg's output and dynamically decides what to do next. If the implement step partially succeeds, it could split the remaining work into parallel flight legs.
2. **Multi-agent orchestration** — launch multiple Claude sessions in parallel for independent plan items, then merge results. A 10-task plan could complete in 3 flight legs instead of 10.
3. **Learning from history** — track which plans succeed/fail by pattern, and pre-emptively adjust skill parameters (e.g., "plans touching database migrations have a 60% first-attempt failure rate — add extra context to the implement prompt").
4. **Cost-aware routing** — dynamically choose between Sonnet (fast/cheap) for straightforward tasks and Opus (thorough) for complex ones.

The current plan (sequential execution with fixed skills) is the right foundation — it's the "bicycle" that must work before building the "car." But the architecture should not preclude the 10x version.

## Accepted Scope (added to this plan)

- **Plan validation:** At workflow start, validate that all configured skill names resolve to actual skills. Return clear error listing invalid skills before any Claude sessions launch.
- **Smart retry:** When a flight leg fails, capture the error output and include it as context in the retry prompt. The resume skill gets "Previous attempt failed with: {error}. Please address this and continue."
- **Autopilot history:** New `boss autopilot list` CLI command showing past/active workflows with status, flight leg count, duration, and outcome. Reads from the workflows table.
- **Diff preview:** After the verify step succeeds, show a summary of all files changed (git diff --stat against the base branch) before proceeding to the land step. Optionally pause for user confirmation.

## Terminology

- **Flight leg:** One skill execution in the autopilot sequence (e.g., running boss-implement). Each flight leg creates one Claude **attempt** via `HostService.CreateAttempt`.
- **Workflow:** The full end-to-end autopilot execution from plan to PR. Persisted in the `workflows` table.
- **Attempt:** A single Claude CLI invocation within a flight leg. Tracked via `HostService.GetAttemptStatus` polling.

## Workflow State Model

```
workflows table:
  id            TEXT PRIMARY KEY
  session_id    TEXT NOT NULL (FK → sessions)
  repo_id       TEXT NOT NULL (FK → repos)
  plan_path     TEXT NOT NULL
  status        TEXT NOT NULL (pending|running|paused|completed|failed|cancelled)
  current_step  TEXT NOT NULL (plan|implement|handoff|resume|verify|land)
  flight_leg    INTEGER NOT NULL DEFAULT 0
  max_legs      INTEGER NOT NULL DEFAULT 20
  last_error    TEXT
  config_json   TEXT (serialized AutopilotConfig overrides)
  created_at    DATETIME NOT NULL
  updated_at    DATETIME NOT NULL

State transitions:
  pending → running     (StartWorkflow RPC)
  running → paused      (flight leg fails, or user pauses)
  running → completed   (land step succeeds)
  running → cancelled   (user cancels via CLI)
  paused  → running     (user resumes via CLI)
  paused  → cancelled   (user cancels while paused)
  *       → failed      (unrecoverable error, e.g., missing repo)
```

## CLI Interface

```
boss autopilot start <plan-file> [--max-legs N] [--confirm-land]
  Start a new autopilot workflow for the given plan file.
  --max-legs: override default max flight legs (default: 20)
  --confirm-land: pause for user confirmation before landing

boss autopilot status [workflow-id]
  Show status of active workflow (or specific workflow by ID).

boss autopilot list [--all]
  List recent workflows. --all includes completed/cancelled.

boss autopilot pause [workflow-id]
  Gracefully pause after current flight leg completes.

boss autopilot resume [workflow-id]
  Resume a paused workflow from where it left off.

boss autopilot cancel [workflow-id]
  Cancel a workflow. If a flight leg is running, waits for it to finish
  then marks workflow as cancelled.
```

In-chat trigger: `/autopilot <plan-file>` within any active session. Prompts for confirmation, then hands off to the autopilot plugin.

## Error Recovery Policy

- **Auto-retry:** On flight leg failure, retry once (2 total attempts per step). Error context injected into the resume prompt ("Previous attempt failed with: {error}"). If error is non-actionable (timeout, crash, empty), use generic retry prompt.
- **Pause trigger:** After the retry also fails (2 consecutive failures on the same step), pause the workflow and notify the user.
- **Non-retryable errors:** Missing repo, invalid skill, plugin crash — pause immediately, no retry.
- **Cancel vs pause:** `cancel` is terminal (workflow cannot be resumed). `pause` preserves state for later resume.

## Baseline Scope (from eng review)

The following architectural decisions form the baseline, resolved during eng review:

1. New `WorkflowService` proto service type
2. `HostService.CreateAttempt` + polling (5s interval) for attempt status only. Plugin-level task polling (TaskSource) is separate and per-plugin.
3. New `workflows` table in daemon SQLite for persistent state (see schema above)
4. Rename existing skills to `boss-` prefix. System skills (boss-plan, boss-implement, etc.) are bundled. Users can override any skill via `AutopilotConfig.skills` struct.
5. Both `boss autopilot` CLI subcommand + in-chat trigger (see CLI Interface above)
6. Watch configurable handoff directory (default `docs/handoffs/`). After each flight leg, scan for files with mtime > attempt start time. Store last processed handoff path in workflow state.
7. New lightweight `AttemptRunner` in `bossd/internal/autopilot/runner.go` (~50 lines). Thin wrapper around `RunClaude()` that adds workflow context (flight leg number, step name) to attempt metadata.
8. Shared `RunClaude(ctx, dir, prompt)` function in `bossd/internal/claude/runner.go` (DRY with SessionCreator)
9. Top-level `AutopilotConfig` typed struct in `lib/bossalib/config/config.go` under `settings.json → autopilot`. Fields: skills (AutopilotSkills), handoff_dir, poll_interval_seconds, max_flight_legs, confirm_land.
10. Validate skill names at workflow start
11. Table-driven tests (38+ cases)
12. Pause + notify on failure strategy (see Error Recovery Policy above)
13. 5-second poll interval for attempt status
14. Real-time output streaming via gRPC server-streaming RPC (`HostService.StreamAttemptOutput`). Requires buffered channel + goroutine per stream + cleanup on disconnect (~150-200 lines). Consumed by `boss autopilot status --follow`.

## Reviewer Concerns

The following implementation-level details were flagged during spec review and should be resolved during implementation planning (not this scope document):

1. **Handoff file format** — What format/content do handoff files contain? How are they created by skills? (Resolve when implementing boss-handoff skill)
2. **Workflow ID generation** — UUID vs timestamp-based. (Implementation decision — use UUID for simplicity)
3. **Session lifecycle coupling** — What happens when the parent session is deleted while a workflow is running? (Resolve: workflow continues independently; session_id is for attribution, not lifecycle coupling)
4. **Confirmation flow mechanics** — Does `--confirm-land` pause the workflow (requiring `resume`) or block a CLI connection? (Resolve: workflow pauses with status=paused, user runs `boss autopilot resume`)
5. **In-chat trigger handoff** — `/autopilot` triggers `HostService.StartWorkflow` RPC, displays "Workflow started, use `boss autopilot status` to monitor", chat session continues normally
6. **Default workflow-id resolution** — When no workflow-id is provided, CLI operates on the most recent active/paused workflow; errors if multiple are in-flight
