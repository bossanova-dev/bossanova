## Problem Statement

Boss is a Claude Code orchestrator with a TUI client, local daemon (bossd), cloud orchestrator (bosso), and web frontend. The founder runs 5-6 Linear tasks in parallel across multiple companies and burns significant daily time on:

1. **Dependabot toil** — manually rebase-merging dependabot PRs one by one across multiple projects
2. **Last-mile fix loops** — responding to AI code reviews, fixing lint, pushing, waiting for CI, getting new review comments, repeating. The infrastructure exists (12-state machine, FixLoop, Dispatcher) but doesn't close the loop end-to-end without human intervention
3. **Attention routing** — no way to see which of 5-6 parallel sessions is blocked on the human vs. still working autonomously. Context-switching to check each one is expensive
4. **Tech debt entropy** — no scheduled time or automation to address it, so it compounds indefinitely

## Demand Evidence

The founder is the primary user — dogfooding daily across multiple projects and companies. The pain is specific: "I burn a lot of time at the end responding to AI code reviews, fixing linting, pushing, waiting, doing it again as new reviews find issues." Dependabot merges are pure mechanical toil repeated across repos. Tech debt "piles up and is rarely addressed" because there's no trigger to do it.

The broader market signal: AI-assisted coding is scaling rapidly, but orchestration tooling hasn't caught up. Developers running multiple Claude/Copilot sessions in parallel have no unified control plane. The "growing wave" of developers hitting this workflow ceiling is real but early.

## Status Quo

Today's manual workflow:

- **Dependabot:** Open GitHub, navigate to each repo, review each dependabot PR individually, click rebase-merge. Repeat across N projects.
- **Fix loops:** Claude implements a task → pushes → CI runs → review comments arrive → founder reads comments, decides what to tell Claude, types instructions, waits for next push. Multiple round trips per PR.
- **Task routing:** Founder manually checks each Linear board and each Boss session to figure out which one needs attention. No aggregated "blocked on me" view.
- **Tech debt:** Doesn't happen. No scheduled time, no automated triage. Accumulates until something breaks.

## Target User & Narrowest Wedge

**Target user:** The founder — a senior developer running AI coding agents in parallel across multiple repos/companies. Uses Claude Code daily, manages work in Linear and GitHub Issues, needs the agent to be more autonomous in the last mile.

**Narrowest wedge:** The end-to-end fix loop automation. The infrastructure already exists in bossd (state machine, FixLoop, Dispatcher, Poller) but the automation flags aren't wired up and the loop doesn't fully close without human intervention. Completing this in the open source core — making `can_auto_address_reviews`, `can_auto_resolve_conflicts`, and `can_auto_merge_dependabot` actually work — delivers immediate value with minimal new architecture. The plugin system then extends from this foundation.

## Constraints

1. **Trust boundary:** An agent writing code on your repo must be open source. boss, bossd, and bossalib are already MIT-licensed. This is non-negotiable for adoption.
2. **Local-first:** Must work on a developer's machine without cloud dependency. Cloud (bosso) is additive, not required.
3. **Plugin isolation:** Plugins must not crash the daemon. Process-level isolation via go-plugin.
4. **Existing architecture:** Must build on the current patterns — VCS Provider abstraction, ConnectRPC, protobuf, interface-based DI, Bubbletea TUI.
5. **Monetization:** Open source core (boss + bossd + bossalib) + paid plugins (scheduler, task trackers, GitHub App webhooks).

## Premises

1. The last-mile fix loop is the highest-leverage problem to solve next — agreed.
2. A scheduler is a new subsystem that creates sessions on the user's behalf, distinct from user-initiated sessions — agreed.
3. The attention routing problem is solvable with better status visibility, not new architecture — agreed.
4. Task tracker integration (Linear + GitHub Issues) is the wedge that makes this a product, not just a personal tool — agreed.
5. The plugin boundary belongs in bossd, with paid features as separate go-plugin binaries — agreed.
6. Developers won't trust a closed-source agent working on their code; open-core is the right business model — agreed.

## Approaches Considered

### Approach A: Local-First — Extend bossd Directly

Add scheduler, task tracker pollers, and expanded webhook handling directly into bossd. Everything compiled into one binary. No plugin system.

**Effort:** M (2-3 weeks human / ~3 hours CC)
**Risk:** Low
**Pros:** Fastest to dogfood; no plugin complexity; single binary deployment
**Cons:** No open-core boundary; paid features compiled into open source binary; harder to evolve into a product; users can't extend it

### Approach B: Cloud-Hub — Extend bosso

Make bosso the automation brain. Receives webhooks, runs scheduled jobs, dispatches to daemons.

**Effort:** L (4-5 weeks human / ~5 hours CC)
**Risk:** Medium
**Pros:** Product-grade; works 24/7; web dashboard; natural multi-user path
**Cons:** Requires cloud infra running for any automation; slower to dogfood; doesn't solve the open source trust problem for the automation layer

### Approach C: Open-Core with go-plugin Boundary in bossd (CHOSEN)

Plugin host in bossd using hashicorp/go-plugin. Plugin interfaces defined as protobuf services. Paid features ship as separate binaries.

**Effort:** L (5-6 weeks human / ~6 hours CC)
**Risk:** Medium
**Pros:** Clean open-core boundary; process isolation; cross-language plugin support; follows existing protobuf patterns; extensible by users and third parties; battle-tested library (Terraform/Vault/Nomad)
**Cons:** Most upfront work; plugin interface design requires care; go-plugin adds a dependency

## Recommended Approach

**Approach C: Open-Core with go-plugin Boundary in bossd.**

The plugin boundary IS the business boundary. The open source core is a genuinely useful local Claude Code orchestrator. Paid plugins add automation, scheduling, and task tracker integration. go-plugin provides process isolation, gRPC communication (matching existing protobuf patterns), and a proven model for exactly this kind of extensibility.

### Plugin Interface Definitions (Proto Services)

Four plugin interface types, defined in `proto/bossanova/v1/plugin.proto`:

**1. TaskSource — Pull tasks from external trackers**

```protobuf
service TaskSourcePlugin {
  // List tasks available for automation (filtered by assignee, labels, status)
  rpc ListAvailableTasks(ListAvailableTasksRequest) returns (ListAvailableTasksResponse);
  // Claim a task (mark as in-progress, assign to this daemon)
  rpc ClaimTask(ClaimTaskRequest) returns (ClaimTaskResponse);
  // Sync Boss session state back to the tracker (started, PR opened, merged, blocked)
  rpc SyncTaskStatus(SyncTaskStatusRequest) returns (SyncTaskStatusResponse);
  // Get task details (title, description, acceptance criteria → Claude prompt)
  rpc GetTaskDetail(GetTaskDetailRequest) returns (GetTaskDetailResponse);
}
```

Implementations: `bossd-plugin-linear`, `bossd-plugin-github-issues`

**2. EventSource — Stream events into the Dispatcher**

```protobuf
service EventSourcePlugin {
  // Start streaming events (webhook-driven or poll-driven)
  rpc Subscribe(SubscribeRequest) returns (stream EventNotification);
  // Configure the event source (webhook URL, poll interval, filters)
  rpc Configure(ConfigureRequest) returns (ConfigureResponse);
}

message EventNotification {
  string repo_id = 1;
  oneof event {
    DependabotPRReady dependabot_pr_ready = 2;
    ReviewReceived review_received = 3;
    ChecksCompleted checks_completed = 4;
    IssueAssigned issue_assigned = 5;
    PRCommentAdded pr_comment_added = 6;
  }
}
```

Implementations: `bossd-plugin-github-app` (webhooks), enhanced VCS poller

**3. Scheduler — Time-based triggers**

```protobuf
service SchedulerPlugin {
  // Register a recurring job (cron expression + action definition)
  rpc RegisterJob(RegisterJobRequest) returns (RegisterJobResponse);
  // List all registered jobs and their next run times
  rpc ListJobs(ListJobsRequest) returns (ListJobsResponse);
  // Manually trigger a job
  rpc TriggerJob(TriggerJobRequest) returns (TriggerJobResponse);
}

message JobAction {
  oneof action {
    CreateSessionAction create_session = 1;    // spawn Claude with a prompt
    AutoMergeAction auto_merge = 2;            // batch-merge matching PRs
    RunScriptAction run_script = 3;            // execute a shell script
  }
}
```

Implementation: `bossd-plugin-scheduler`

**4. NeedsAttention — Attention routing (in core, not a plugin)**
This lives in the open source core, not as a plugin. It aggregates signals from all sources:

```protobuf
enum AttentionReason {
  ATTENTION_REASON_UNSPECIFIED = 0;
  BLOCKED_MAX_ATTEMPTS = 1;         // fix loop exhausted, needs human
  AWAITING_HUMAN_INPUT = 2;         // Claude asked a question
  REVIEW_REQUESTED = 3;             // PR ready for human review
  MERGE_CONFLICT_UNRESOLVABLE = 4;  // auto-resolve failed
  TASK_CLARIFICATION_NEEDED = 5;    // task description insufficient
}

message AttentionStatus {
  bool needs_attention = 1;
  AttentionReason reason = 2;
  string summary = 3;
  google.protobuf.Timestamp since = 4;
}
```

Hydrated into `ListSessions` responses. Surfaced in TUI with a distinct column/indicator. Sortable so "needs me" sessions float to the top.

### Plugin Host Design

**Discovery:** Plugins configured in `settings.json`:

```json
{
  "plugins": [
    {
      "name": "scheduler",
      "path": "/usr/local/bin/bossd-plugin-scheduler",
      "enabled": true,
      "config": { "timezone": "America/New_York" }
    },
    {
      "name": "linear",
      "path": "/usr/local/bin/bossd-plugin-linear",
      "enabled": true,
      "config": { "api_key_env": "LINEAR_API_KEY", "team_id": "TEAM-123" }
    }
  ]
}
```

**Lifecycle:** bossd starts plugins on daemon startup, health-checks them periodically, restarts on crash. Plugins get a bootstrap context with: repo list, session list, event bus handle.

**Event bus:** Internal pub/sub in bossd. Plugins publish events (task available, webhook received, cron fired). The Dispatcher subscribes and routes them into the existing session state machine. The event bus is the integration seam — plugins only interact with the core through events and RPCs.

**Per-repo plugin config:** Extend the existing automation flags pattern. Each plugin can define per-repo settings that appear in the TUI repo settings screen:

```
┌─ Repo Settings: bossanova ──────────────────────────────┐
│ Auto-merge passing PRs:         [OFF]                   │
│ Auto-merge Dependabot:          [ON]                    │
│ Auto-address review comments:   [ON]                    │
│ Auto-resolve merge conflicts:   [ON]                    │
│                                                         │
│ ── Scheduler Plugin ──                                  │
│ Tech debt sweep:                [Weekly, Sunday 2am]    │
│ Dependency update:              [Daily, 6am]            │
│                                                         │
│ ── Linear Plugin ──                                     │
│ Sync tasks from:                [Team: Platform]        │
│ Auto-start priority:            [P0 and P1 only]        │
└─────────────────────────────────────────────────────────┘
```

### Implementation Phases

**Phase 1: Wire up existing automation flags (open source core)**

- Make `can_auto_merge_dependabot`, `can_auto_address_reviews`, `can_auto_resolve_conflicts` actually drive behavior in the Dispatcher and FixLoop
- This delivers immediate value with zero new architecture
- Add `AttentionStatus` to proto and hydrate in `ListSessions`
- Add attention column to TUI home screen

**Phase 2: Plugin host infrastructure**

- Add `hashicorp/go-plugin` dependency
- Define plugin proto interfaces in `proto/bossanova/v1/plugin.proto`
- Build plugin host in `services/bossd/internal/plugin/` — discovery, lifecycle, health
- Build event bus in `services/bossd/internal/plugin/eventbus/`
- Add plugin config to `settings.json` schema

**Phase 3: Scheduler plugin**

- `bossd-plugin-scheduler` as a separate Go module/binary
- Cron expression parser (robfig/cron)
- Job types: create session, auto-merge batch, run script
- Per-repo schedule config via plugin settings RPC
- First job: "dependabot batch-merge" — find all passing dependabot PRs across repos, merge them

**Phase 4: Task source plugins**

- `bossd-plugin-linear` — Linear GraphQL API, poll for assigned tasks, map to session creation, sync status back
- `bossd-plugin-github-issues` — GitHub Issues API, same pattern
- Task → session mapping: task title/description becomes Claude prompt, task labels become session metadata
- Bidirectional sync: session state changes update task status in tracker

**Phase 5: Enhanced GitHub integration**

- `bossd-plugin-github-app` — GitHub App webhook receiver (local mode via smee.io/ngrok, cloud mode via bosso relay)
- Real-time events instead of polling
- Richer event set: PR reviews, check runs, issue comments, dependabot alerts

## Design Decisions (Resolved)

1. **Plugin distribution:** Auth0 login validates entitlement at plugin startup. Discovery via marketplace UI (future). Homebrew tap and Go binary downloads as distribution channels.
2. **Plugin marketplace:** Not for now. Architecture should allow third-party plugins in the future, but initial plugins are first-party only.
3. **Event bus persistence:** In-memory only for now. Events are ephemeral — no replay after daemon restart. Can add SQLite persistence later if needed.
4. **Cloud relay for webhooks:** `relay.bossanova.dev` — a first-party relay service. Part of the orchestrator layer (may be a separate service or integrated into bosso). No dependency on smee.io or ngrok.
5. **Plugin versioning:** Semver for plugin proto interfaces. go-plugin protocol versioning enforces compatibility at startup.
6. **Task source priority:** Linear takes priority over GitHub Issues. When both plugins are active, Linear tasks are processed first. Deduplication by external task ID — if a Linear task references a GitHub Issue, the Linear version wins.

## Success Criteria

1. **Fix loop closes autonomously:** A PR with failing checks or review comments triggers automated Claude fix attempts without human intervention, up to the configured max attempts. When max attempts are reached, the session surfaces as "needs attention."
2. **Dependabot PRs merge themselves:** Passing dependabot PRs across all registered repos are batch-merged on a schedule without human intervention.
3. **"Needs attention" is visible at a glance:** TUI home screen shows which sessions need the human, why, and how long they've been waiting. Sessions needing attention sort to the top.
4. **Linear tasks become sessions automatically:** A task assigned to the user in Linear appears as a Boss session within the configured poll interval, with Claude working on it using the task description as the prompt.
5. **Tech debt gets addressed on schedule:** A weekly cron job creates sessions that address configured tech debt categories (lint fixes, dependency updates, dead code removal).
6. **Plugin system is extensible:** A developer can write a custom plugin implementing one of the four proto interfaces and load it into their bossd instance.
